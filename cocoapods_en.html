How to publish your first Pod, step by step

Spanish Version Here

The main reason of this post is that during the last weeks I have been preparing a pod to upload to CococaPods, and I could see that there is no documentation for newbies. But thanks to the help of great people like Dani García (@fillito) I could finally publish it.

To help people that could have been in the same situation as me, I am launching this little tutorial on where I will explain the way that I used.

Preparing the ground

There are a lot of ways to do a Pod and, knowing that, I am going to explain the way that could be the easiest.

Let's go to Github and create a new repo. Before you create it, mark the README and LICENSE options. If you already have a prepared README and License, forget this step.

When we have our repo created, use your Git client to clonate it in your computer. Henceforth, during this post, we are going to call MainFolderthe folder that contains the repo on the computer.

At the moment we have all the files locally, we are going to create a new XCode project. This project will be the Demoof our Pod. In this project we should show the things that it does in a simple and clear way.

Recomendation:
Dani García recommended me to create this project in a directory outside the one
 that we are working on, rename the containing folder to Demo and put that
  folder inside the folder we are working on. Somethong like that:

If the folder we are working on is located in /Documents/MainFolder, we are
 going to create the XCode project on the desktop, for example. Then we 
 would have /Desktop/MiPodProject

Then we should rename the "MiPodProject" folder to "Demo", and move it 
to /Documents/MainFolder

/MainFolder/
     README.md
     License.md
     /Demo
        /...

The finality of that is to avoid creating a project called Demo, and that
 all our classes have the Demo name inlaid but the name that we want. 
 This is a point of style, so you can ignore it if you don't find it useful

Once we have our project, we have to make a new folder in our MainFolder, this one is going to contaon the files that are going the Pod itself, the only ones that are going to be dowloaded to the user. Henceforth we are going to call that folder PodContainer.

  • /CarpetaPrincipal/
    • README.md
    • License.md
    • /Demo
      • /...
    • /PodContainer
      • /... <--- Files that are going to be uploaded/downloaded

Now that we have our files on its place, we are going to move to the Demo project. We are going to create a new group, and we call it with the same name as our Pod. Now we have to drag the content of PodContainer inside this group, and we choose this options:

That way, we are going to use the /PodContainerfiles inside our Demo project, but without having to make them to be inside the project, we have them "linked". We can work with them, and all the changes are going to be saved in the original files.

Finally, but not less important, we are going to create the *.podspec. This file is going to manage the configuration of our Pod. You can download the following file, which is going to be the same as the example I am going to show:

Download

Let's see it in detail:

Pod::Spec.new do |s|
  s.name         = "PQFCustomLoaders"
  s.version      = "0.0.1"
  s.summary      = "Collection of Custom Loaders highly customizable."
  s.homepage     = "https://github.com/poolqf/PQFCustomLoaders"
  s.license      = 'MIT'
  s.author       = "Pol Quintana"
  s.source       = { :git => "https://github.com/poolqf/PQFCustomLoaders.git", :tag => "0.0.1" }
  s.platform     = :ios, '7.0'
  s.source_files = 'PQFCustomLoaders'
  s.frameworks   = 'UIKit'
  s.requires_arc = true
  s.social_media_url = 'https://twitter.com/poolqf'
  s.dependency 'UIColor+FlatColors'
end
  • s.name : Name of the Pod
  • s.version : Actua version
  • s.summary : Small description
  • s.homepage : You can put here the link you want, if you wave a portfolio, put it here.
  • s.license: The license of the Pod
  • s.author : Name of the creator
  • s.source :
    • :git => : Link to the repo where the project is located (The whole folder)
    • :tag => : The tag, in github, that points to the "s.version"
  • s.platform : :ios, '#Minimum iOS version where it is going to work#'
  • s.source_files: Folder, inside the repo, where are the files that are going to be dowloaded to the user to make it work. That way, we are avoiding the whole repo download when someone use your Pod.
  • s.frameworks : Frameworks needed for the Pod
  • s.requires_arc: If the Pod needs ARC
  • s.social_media_url : https://twitter.com/yourUserName
  • s.dependency: Other pods that your Pod need to work.

Now, what we are going to do is to edit it. Then, we have to rename it to be something like that:

  • /MainFolder/
    • README.md
    • License.md
    • NombreDelPod.podspec <---
    • /Demo
      • /...
    • /PodContainer
      • /...

¡DO NOT USE TEXTEDIT TO EDIT THE PODSPEC! My recomendation is XCode, but you can use others.

Pre-upload moment

Before uploadint the Pod, we will have to do more things. First, we are going to check that our *.podspec is correct. Open the Terminal and move to the MainFolder.

Once there, run:

pod lib lint

If everything is correct, we are going to open our Git client and push ONLY the *.podspec, that is now verified.

Now we have in our repo a *.pospec that points to a version(a tag) that is not in github. To solve this, we are going back to the Git client and push all the other files.

And.. ¡¡¡ We are going to add a tag to this commit, that has to be exactly the same as our s.version!!!

Now, the last thing before starting the upload process is to check that not only the *.podspec is correct, but all the repo is ready too. We are going to run:

pod spec lint

Let's publish

Now that we have everything ready, we are only missing a CocoaPods registration. We are going to do it this way:

Move to the MainFolder from the Terminal, and run:

pod trunk register pol.quintana1@gmail.com 'Pol Quintana' --description='macbook pro'

The description is like a computer id that we use to upload the Pod

In just a seconds we are going to recieve a mail to verify that it is correct. Once verified... Let's publish!

To publish, we run:

pod trunk push

That's all, if everything goes well, our Pod will be published
